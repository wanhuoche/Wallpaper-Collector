# 壁纸收集器

一个全栈壁纸搜索引擎，支持多图源聚合搜索，附带用户登录/注册系统。

## 技术栈

### 前端
- 纯静态 HTML + CSS + JS（IIFE 模块），无框架无构建工具
- 通过 IndexedDB + File System Access API 持久化设置
- Apple 风格 UI（SF Pro 字体，毛玻璃导航栏，圆角卡片）

### 后端
- Cloudflare Pages Functions（`functions/`），零 npm 依赖
- Cloudflare D1（SQLite 兼容）数据库
- Web Crypto API 实现 PBKDF2 密码哈希 + HMAC-SHA256 JWT

## 功能

### 用户系统
- 邮箱 + 密码注册 / 登录
- JWT 认证（7 天过期）
- 导航栏头像圆圈（取用户名首字）+ 下拉菜单退出
- 收藏夹云端同步（登录后自动双向合并，按收藏时间取最新）
- 设置云同步（登录后自动拉取/推送设置到云端）
- 未登录时可正常使用搜索功能，登录按钮入口
- 后端未启动时优雅降级，不影响核心搜索功能

### 游客模式
- 无需注册、无需 API Key 即可搜索（后端代理转发，Key 不出服务器）
- 游客（未登录）：每日 20 次免费搜索
- 已登录但未填自己的 Key：每日 40 次搜索
- 填写自己的 API Key 后：无限制，浏览器直连 API
- 搜索结果统计栏实时显示剩余次数
- 超额自动提示"今日次数已用完，请填写自己的 API Key 或明天再试"
- 未登录时收藏按钮跳转登录页（不写入 localStorage）
- 退出登录自动清除本地收藏夹数据

### 搜索
- 输入关键词搜索壁纸
- 支持分页加载（"加载更多"按钮 + 无限滚动）
- 三个维度的筛选标签：

## 架构说明

### 前端模块化

所有前端 JS 采用 IIFE 模式（`(function() { ... })()`），通过 `window.WallpaperApp` 全局命名空间共享状态、DOM 引用和 API。

加载顺序：`theme.js`（head 优先）→ `source-config.js` → `search.js` → `storage.js` → `auth.js` → `app.js` → `favorites.js`

### 图源配置模式（`SOURCE_CONFIG`）

每个图源实现一组标准接口：

- `name` — 显示名称
- `baseUrl` — API 根地址
- `buildParams(query, perPage, page, opts)` — 构建请求参数
- `getAuthHeader(apiKey)` — 返回认证头
- `parseResponse(data)` — 解析 API 响应为统一格式 `{ total, photos[] }`
- `mapRatio(selected)` — 比例筛选 → 图源特定参数

这种设计使得添加新图源只需要在 `SOURCE_CONFIG` 中增加一个条目。

### 数据流

```
用户输入 → 翻译（如需）→ 构建 API 请求 → fetch → 解析响应
         → 前端比例筛选 → 前端清晰度筛选 → 渲染网格
```

### 状态管理

- 全局 `state` 对象保存运行时状态（含 `user` 登录用户信息）
- IndexedDB + File System Access API 保存持久化设置
- JWT Token 存 localStorage，启动时验证
- 搜索结果 `state.allPhotos` 跨分页累积（上限 500 张）

## Bug 修复记录

### 1. Wallhaven 下载跳转新标签页而非触发浏览器下载

- **现象**：Wallhaven 图源点击下载按钮后，图片在新标签页打开，没有触发浏览器下载；Pixabay / Unsplash 正常
- **根因**：Wallhaven 图片 CDN（`w.wallhaven.cc` 等）设置了严格的 CORS 策略，禁止跨域 `fetch`。下载函数通过 `fetch(url)` 获取 blob 再创建 `blob:` URL 触发下载，CORS 被拒后抛异常，进入 `catch` 分支执行 `window.open(url, '_blank')`
- **修复**：`search.js` 的 `downloadPhoto()` 加入双 URL 重试机制 — 直连 `fetch` 失败后，改为通过 Cloudflare Worker 代理（`worker_url?url=图片URL`）再 fetch 一次。Worker 返回带 `Access-Control-Allow-Origin` 头的响应，浏览器允许读取 blob，成功触发下载。两次都失败才 fallback 到新标签页
- **涉及文件**：`js/search.js`

### 2. API Key 重启后丢失

- **现象**：`file://` 协议打开的本地 HTML，重启电脑后浏览器清空 `localStorage`，API Key 和设置全部丢失
- **根因**：`localStorage` 数据跟随浏览器，不跟随项目文件，浏览器可能在系统重启时清理 `file://` 来源的存储
- **修复**：三层存储方案 — IndexedDB（日常读写，更不易被清理）→ File System Access API（用户选目录后自动写入磁盘 JSON，重启优先从文件恢复）→ 手动导入/导出 JSON（全浏览器兜底）。`localStorage` 仅保留旧数据迁移逻辑，不再写入
- **涉及文件**：`js/storage.js`

### 3. 快速切换筛选标签导致请求堆积

- **现象**：连续点击多个比例/清晰度标签时，每个标签变更都发起一个请求，旧请求的响应可能覆盖新结果
- **根因**：筛选标签 click 事件直接调用 `doSearch()`，无防抖无取消机制
- **修复**：标签变更加入 300ms debounce（`app.js` 中 `filterSearch()`），同时 `doSearch()` 在发起新请求前通过 `AbortController.abort()` 取消上一个未完成的 fetch，被取消的请求在 catch 中按 `err.name === 'AbortError'` 静默退出
- **涉及文件**：`js/app.js`、`js/search.js`

### 4. 错误提示模糊

- **现象**：网络断开、API 限流、密钥过期等所有错误统一显示"请求失败 (xxx)"，用户无法判断原因和解决方式
- **修复**：`search.js` 的 `doSearch()` 中，`fetch` 返回非 2xx 时按 HTTP 状态码分类提示 — 401 "API Key 无效或已过期"、403 "API 拒绝访问"、429 "请求过于频繁，请稍后重试"、5xx "图源服务器故障"。同时检测 `fetch` 完全失败（`TypeError: Failed to fetch`）并提示"网络连接失败"
- **涉及文件**：`js/search.js`

### 5. 收藏夹切回搜索结果后图片不消失

- **现象**：首次打开页面未搜索时切换到收藏夹，再切回搜索结果标签，收藏夹的图片仍然显示在页面上
- **根因**：切换标签时通过 `_searchGridHTML` 保存/恢复搜索结果网格的 HTML。初始未搜索时网格为空（`innerHTML` 为空字符串 `''`），恢复时用 `if (W.state._searchGridHTML)` 判断，空字符串被当作 falsy 跳过，导致未执行 `resultsGrid.innerHTML = ...` 恢复操作
- **修复**：将判断条件从 `if (W.state._searchGridHTML)` 改为 `if (typeof W.state._searchGridHTML === 'string')`，空字符串也是 string 类型，不会被跳过
- **涉及文件**：`js/favorites.js`

### 6. 预览弹窗竖长图显示错误 + 渲染慢

- **现象**：竖长比例原图（如 3409×5113）在预览弹窗中被按 16:9 裁剪显示，只能看到局部；即使修复后加载原图，5-20MB 的文件下载太慢
- **根因**：弹窗加载的是 `thumbs.large`（Wallhaven 强制 16:9 裁剪的缩略图，432×243），浏览器按缩略图文件的实际宽高比渲染 `<img>` 元素；改用原图后又因文件太大导致加载缓慢。且 `w.wallhaven.cc` 有 Referer 防盗链，直接请求返回 403
- **修复**（三层优化）：
  1. **比例修正**：JS 设置 `img.style.aspectRatio = photo.width + ' / ' + photo.height`，浏览器按原图尺寸计算宽高比，不受缩略图文件实际比例影响
  2. **防盗链**：`<img>` 设 `referrerPolicy = 'no-referrer'`，浏览器不发送 Referer 头，绕过 Wallhaven CDN 的 403 限制
  3. **秒开**：新增 `photo.medium` 字段，用 `thumbs.original`（不裁剪中等尺寸缩略图，最长 1920px，保持原比例），弹窗优先加载此字段；CSS `max-height: calc(96vh - 110px)` 给竖图留足纵向空间
  4. **预加载**：打开弹窗时后台 `new Image()` 预加载前后各 2 张图到浏览器缓存，切图瞬间显示
  5. **查看原图**：弹窗底部新增"查看原图"按钮，点击加载 `photo.full` 全尺寸图片，按钮有加载状态反馈
- **涉及文件**：`js/search.js`、`js/source-config.js`、`style.css`、`js/app.js`

### 7. `updateNavUser()` 事件监听器泄漏

- **现象**：`updateNavUser()` 每次调用都通过 `document.addEventListener('click', ...)` 添加匿名函数来关闭用户下拉菜单，但从不移除。多次调用（如反复登录/登出）会累积多个同功能监听器，造成内存泄漏和重复响应
- **根因**：使用匿名函数注册 document click 事件，无法在下次调用时通过 `removeEventListener` 移除
- **修复**：将匿名函数提取为命名函数 `closeUserDropdown(e)`（通过 `navUser.contains(e.target)` 判断点击是否在下拉菜单外），调用 `updateNavUser` 时先 `document.removeEventListener('click', closeUserDropdown)` 再重新注册
- **涉及文件**：`js/auth.js`

### 8. login / register 表单逻辑重复

- **现象**：`login.html` 和 `register.html` 的内联脚本中，表单提交处理逻辑几乎完全一致（loading 态切换、错误显示、成功后跳转），后续加"记住我"或第三方登录需改两处
- **修复**：在 `js/auth.js` 中新增 `handleFormSubmit(form, action)` 方法，自动读取表单内 `.auth-input` 的 `name` + `value` 组装字段对象，统一处理 loading/错误/跳转逻辑。`login.html` 和 `register.html` 的内联脚本各缩减为 ~5 行，分别传入 `W.auth.login` 和 `W.auth.register` 作为 action
- **涉及文件**：`js/auth.js`、`login.html`、`register.html`

### 9. `API_BASE` 硬编码

- **现象**：`js/auth.js` 中后端地址写死为 `http://localhost:3000`，部署到其他环境或修改端口需要改源码
- **修复**：在 `index.html`、`login.html`、`register.html` 的 `<head>` 中添加 `<meta name="api-base" content="http://localhost:3000">`，`auth.js` 启动时读取该 meta 标签，未配置时 fallback 到默认值
- **涉及文件**：`js/auth.js`、`index.html`、`login.html`、`register.html`

### 10. bcrypt 同步调用阻塞事件循环 + 密码无上限

- **现象**：`routes/auth.js` 的注册/登录/改密码接口使用 `bcrypt.hashSync()` 和 `bcrypt.compareSync()` 同步方法，并发请求时阻塞 Node 事件循环；密码仅校验 ≥6 位，无上限，恶意长字符串可消耗大量 CPU
- **修复**：
  1. 所有 `hashSync` / `compareSync` 改为 `await bcrypt.hash()` / `await bcrypt.compare()`，路由 handler 改为 `async`
  2. 密码校验改为 `8-16 位`
- **涉及文件**：`server/routes/auth.js`

### 11. Cloudflare Pages Functions 无法解析 npm 依赖

- **现象**：将 API 部署为 Cloudflare Pages Functions 时，`functions/api/[[route]].js` 中的 `import bcrypt from 'bcryptjs'` 和 `import { SignJWT, jwtVerify } from 'jose'` 报错 `Could not resolve "bcryptjs"`、`Could not resolve "jose"`，构建失败。即使添加 `functions/package.json` 或根 `package.json` 也无法解决
- **根因**：Cloudflare Pages 的 Git 自动部署使用内置的 Wrangler v3 进行 Functions 打包。该版本的打包器默认**不会运行 `npm install`** 来安装函数依赖，只会内联解析 `functions/` 目录内的文件。Wrangler v3 的设计假设依赖已在部署前通过本地构建工具（如 `wrangler pages deploy`）预先打包好，与 Git 集成模式的工作流不兼容
- **修复**：彻底去掉 npm 依赖，改用 **Web Crypto API** 原生实现：
  1. **密码哈希** — 用 `crypto.subtle.deriveBits()` + PBKDF2 + SHA-256（10 万次迭代，Workers 上限）
  2. **JWT** — 用 `crypto.subtle.sign()` + HMAC-SHA256 签发和校验 token
  3. **Base64URL** — 手写编解码（Web Crypto 的 JWT 生态中没有内置）
  4. 最终 `functions/api/[[route]].js` 变成**零 import 语句**，全部靠 Workers 运行时原生 API
  5. 数据格式兼容：密码存储格式 `salt:hash`（均为 base64url），JWT 格式与 `jose` 库一致
- **涉及文件**：`functions/api/[[route]].js`
- **教训**：Cloudflare Pages Git 集成 + Functions 的组合中，**应避免 npm 依赖**。需用密码学方案时，优先考虑 `crypto.subtle`（Web Crypto），它是 Workers/Pages 运行时的一等公民，无需任何构建步骤
- **补充**：上线后注册报 500，Function 日志显示 `Pbkdf2 failed: iteration counts above 100000 are not supported`。Workers 运行时对 PBKDF2 迭代次数有 10 万次硬上限（标准 Web Crypto 无此限制），将迭代数从 60 万降至 10 万解决。10 万次 PBKDF2-SHA256 安全性仍足够

### 12. 游客模式选择"安全"仍出现 NSFW 内容

- **现象**：游客模式下，纯度标签选择"安全 (SFW + Sketchy)"，搜索结果仍包含 NSFW 图片
- **根因**：两个问题叠加 —（1）`W.state.selectedPurity` 从存储恢复后（如之前保存过 `all`），纯度标签 UI 始终保持 HTML 默认的"安全"高亮，造成 UI 与状态不一致，用户以为选了安全实际请求带的是 `all`；（2）前端没有本地纯度过滤，完全依赖 API 返回结果
- **修复**：
  1. `app.js` init 时根据 `W.state.selectedPurity` 同步纯度标签的 active 状态
  2. `search.js` 新增 `filterByPurity()`，选择"安全"时前端强制过滤 `purity === 'nsfw'` 的结果，作为安全兜底
  3. 后端 `proxyWallhaven` 纯度条件改为 `purity === 'all' ? '111' : '110'`，消除隐式类型转换风险
- **涉及文件**：`js/app.js`、`js/search.js`、`functions/api/[[route]].js`、`server/routes/guest.js`

### 13. 退出登录后收藏夹未清除

- **现象**：退出登录后以游客身份使用，之前登录态的收藏夹图片仍然显示，且可以继续收藏到 localStorage
- **根因**：`auth.js` 的 `logout()` 仅清除 JWT token，未清除 `localStorage` 中的 `wp_favorites` 数据
- **修复**：`logout()` 中增加 `localStorage.removeItem('wp_favorites')` 和 `W.state.favorites = []`
- **涉及文件**：`js/auth.js`

### 14. 游客模式下点击收藏仍写入 localStorage

- **现象**：未登录时点收藏按钮可以正常收藏，但数据只存在 localStorage，换设备或清浏览器后丢失，体验割裂
- **修复**：搜索结果卡片和预览弹窗两处收藏按钮增加 `W.state.user` 检查，未登录时点击跳转 `login.html`
- **涉及文件**：`js/search.js`、`js/app.js`

### 15. 取消收藏后刷新页面自动恢复

- **现象**：取消收藏某张图片后刷新页面，该图片又出现在收藏夹中；跨设备同步时添加收藏能同步但取消收藏不能
- **根因**：三层"只增不删"：
  1. 服务端 `handleSyncFavorites` 做并集合并（先取云端全部，再合并本地），本地删除的项云端仍保留
  2. 客户端 `mergeLocal()` 同样只做并集（云端有的加到本地，云端没有的本地不动），跨设备删除无法同步
  3. `toggle()` 调用 `syncWithCloud()` 后，该函数用服务端返回的旧数据覆盖本地状态
- **修复**：
  1. 服务端改为**先删后插**（`DELETE` 全部行 → `INSERT` 客户端发来的列表），客户端发送完整收藏列表即为权威数据
  2. 客户端 `mergeLocal()` 改为以云端为基准：本地项在云端缺失且 `savedAt` 不新于云端最新时间 → 丢弃（已在其他设备删除）；离线添加的（`savedAt` 新于云端）予以保留
  3. `toggle()` 改用 `pushFavorites()` 单向推送，不再用 `syncWithCloud()` 覆盖本地
- **涉及文件**：`server/routes/favorites.js`、`functions/api/[[route]].js`、`js/favorites.js`
